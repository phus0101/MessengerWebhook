## Critical Issues

### 🔴 Issue #1: Dependency Injection Lifetime Mismatch

**Root Cause:**
`ConversationMetricsService` registered as **Singleton** but depends on `ITenantContext` which is **Scoped**.

**Evidence:**
```
Error while validating the service descriptor 
'ServiceType: MessengerWebhook.Services.Metrics.IConversationMetricsService 
Lifetime: Singleton 
ImplementationType: MessengerWebhook.Services.Metrics.ConversationMetricsService': 
Cannot consume scoped service 'MessengerWebhook.Services.Tenants.ITenantContext' 
from singleton 'MessengerWebhook.Services.Metrics.IConversationMetricsService'.
```

**Location:**
- `src/MessengerWebhook/Program.cs:266`
- `src/MessengerWebhook/Services/Metrics/ConversationMetricsService.cs:14,24,74`

**Impact:**
- Application cannot start
- All services depending on `IConversationMetricsService` fail validation
- 12 service descriptors affected (WebhookProcessor, QuickReplyHandler, StateMachine, all StateHandlers)

**Why This Violates DI Rules:**
In .NET DI, a singleton lives for the entire application lifetime and is shared across all requests. A scoped service lives for one HTTP request. If a singleton captures a scoped service in its constructor, it will hold onto the FIRST request's scoped instance forever, causing:
- Stale tenant context across requests
- Memory leaks (scoped services never disposed)
- Thread safety issues (scoped services not thread-safe)

## Recommended Fixes

### Fix #1: Remove ITenantContext from Constructor (RECOMMENDED)

**Approach:** Resolve `ITenantContext` per-flush using `IServiceScopeFactory`.

**Implementation:**
```csharp
public class ConversationMetricsService : IConversationMetricsService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<ConversationMetricsService> _logger;
    private readonly ConcurrentQueue<ConversationMetricData> _metricsBuffer;

    public ConversationMetricsService(
        IServiceScopeFactory scopeFactory,
        ILogger<ConversationMetricsService> logger)
    {
        _scopeFactory = scopeFactory;
        _logger = logger;
        _metricsBuffer = new ConcurrentQueue<ConversationMetricData>();
    }

    public async Task FlushAsync(CancellationToken cancellationToken = default)
    {
        // ... dequeue logic ...
        
        using var scope = _scopeFactory.CreateScope();
        var dbContext = scope.ServiceProvider.GetRequiredService<MessengerBotDbContext>();
        var tenantContext = scope.ServiceProvider.GetRequiredService<ITenantContext>();

        var entities = batch.Select(m => new ConversationMetric
        {
            TenantId = tenantContext.TenantId, // ✅ Fresh tenant context per flush
            // ... rest of mapping
        }).ToList();
        
        // ... save logic ...
    }
}
```

**Pros:**
- Minimal code change
- Keeps singleton registration
- Resolves tenant context fresh per flush
- Thread-safe

**Cons:**
- Tenant context resolved during flush, not during log call
- If metrics logged with TenantId in `ConversationMetricData`, this is redundant

---

### Fix #2: Store TenantId in ConversationMetricData (ALTERNATIVE)

**Approach:** Capture `TenantId` when logging, not when flushing.

**Implementation:**
```csharp
// ConversationMetricData.cs
public class ConversationMetricData
{
    public Guid TenantId { get; set; } // ✅ Add this
    // ... existing properties
}

// ConversationMetricsService.cs
public Task LogAsync(ConversationMetricData metricData, CancellationToken cancellationToken = default)
{
    // Caller already has TenantId in metricData
    _metricsBuffer.Enqueue(metricData);
    return Task.CompletedTask;
}

public async Task FlushAsync(CancellationToken cancellationToken = default)
{
    // ... dequeue logic ...
    
    using var scope = _scopeFactory.CreateScope();
    var dbContext = scope.ServiceProvider.GetRequiredService<MessengerBotDbContext>();

    var entities = batch.Select(m => new ConversationMetric
    {
        TenantId = m.TenantId, // ✅ Use TenantId from metric data
        // ... rest of mapping
    }).ToList();
}
```

**Pros:**
- Captures tenant context at log time (more accurate)
- No need to resolve `ITenantContext` in service
- Simpler dependency graph

**Cons:**
- Requires updating all call sites to pass `TenantId`
- Changes `ConversationMetricData` model

---

### Fix #3: Change to Scoped Registration (NOT RECOMMENDED)

**Approach:** Register `IConversationMetricsService` as scoped.

**Why NOT Recommended:**
- Background service `MetricsBackgroundService` is singleton (required for `IHostedService`)
- Would need to resolve metrics service per flush in background service
- Loses in-memory buffer across requests (defeats buffering purpose)
- More complex than Fix #1
